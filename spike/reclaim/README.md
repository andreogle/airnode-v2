# Reclaim Protocol TLS Proof Spike

Research spike to validate TLS proof generation via Reclaim Protocol's attestor.

## Questions this spike answers

1. Can we call an API through the attestor and get a signed proof?
2. Can we redact API keys from the proof?
3. What's the latency overhead?
4. Can we extract specific JSON fields from the proven response?
5. Can the proof be verified off-chain?

## Prerequisites

- Docker (for the attestor)
- Node.js 22+ (attestor-core has native modules incompatible with Bun)

## Setup

```bash
cd spike/reclaim

# Install deps (separate from Airnode's package.json)
bun install --ignore-scripts

# Build the @reclaimprotocol/tls dependency (needs its dev deps)
npm install --prefix node_modules/@reclaimprotocol/tls
npm run --prefix node_modules/@reclaimprotocol/tls build

# Download ZK circuit files (~50MB)
node node_modules/@reclaimprotocol/zk-symmetric-crypto/lib/scripts/download-files.js

# Start the attestor
docker compose up -d

# Verify it's running
curl http://localhost:8001/browser-rpc
# Should print "Attestor Core RPC"
```

## Run

```bash
# Basic: call CoinGecko through the attestor
node --import=./init-crypto.mjs --experimental-strip-types prove.ts

# With API key redaction test
node --import=./init-crypto.mjs --experimental-strip-types prove.ts --redact

# Verify the proof off-chain
node --import=./init-crypto.mjs --experimental-strip-types verify.ts
```

## Files

| File | Purpose |
|------|---------|
| `prove.ts` | Call CoinGecko through attestor, capture proof, measure latency |
| `verify.ts` | Verify proof off-chain (hash check, signature recovery) |
| `init-crypto.mjs` | ESM preload that initializes TLS crypto for attestor-core |
| `docker-compose.yml` | Reclaim attestor container |
| `proof-result.json` | Generated by prove.ts (gitignored) |

## Notes

- This spike uses `@reclaimprotocol/attestor-core` (AGPL-3.0) as a **local dependency only**.
  It is NOT bundled into Airnode. The AGPL boundary is the process boundary.
- Scripts run via Node.js (not Bun) because attestor-core depends on `re2`, a native module
  incompatible with Bun's ABI.
- The `init-crypto.mjs` preload is needed because `@reclaimprotocol/tls` ships with an empty
  crypto object that must be populated before use. The `zk-symmetric-crypto` package has a
  nested copy of `@reclaimprotocol/tls` in its own `node_modules/` — both instances must be
  initialized.
- The attestor's private key in `.env` is Hardhat/Foundry's well-known default account #0.
  Do not use in production.

## Findings

Tested 2026-04-04 on M1 MacBook Pro, Node.js v22.19.0, local Docker attestor.

### Results

| Question | Result | Details |
|----------|--------|---------|
| API call through attestor produces signed proof | **PASS** | Full CoinGecko call, attestor signature verified |
| API key redacted from proof | **PASS** | `secretParams.headers` values not present in proof output |
| Latency (gnark) | **3.48s total** | 0.5s API call + 1.9s ZK proofs (gnark) + 1.0s attestor verification |
| Latency (snarkjs) | **3.85s total** | 0.4s API call + 3.3s ZK proofs (snarkjs) + 0.1s attestor verification |
| Proof verifiable off-chain | **PASS** | Identifier hash matches, ECDSA signature recovery matches attestor address |
| JSON field extraction | **PASS** | `extractedParameters.price: 2064.01` via regex named capture group |

### Proof structure

Reclaim proofs contain:

- **`claim.identifier`**: `keccak256(provider + "\n" + parameters + "\n" + context)`
- **`claim.parameters`**: JSON with URL, method, responseMatches (public — committed to in the identifier)
- **`claim.context`**: JSON with `extractedParameters` (values extracted by the attestor)
- **`signatures.claimSignature`**: EIP-191 personal sign over `identifier + "\n" + owner + "\n" + timestampS + "\n" + epoch`
- **`signatures.attestorAddress`**: the attestor's Ethereum address

### Latency breakdown

```
                            gnark       snarkjs
Connect to attestor:        ~0.00s      ~0.00s
TLS tunnel + API call:      ~0.50s      ~0.35s
ZK proof generation (2x):   ~1.90s      ~3.30s
Attestor verification:      ~1.00s      ~0.12s
Total:                      ~3.48s      ~3.85s
```

gnark (native Go) is the default. Set `ZK_ENGINE=snarkjs` to use the JS backend. The
TLS 1.3 key-update method avoids ZK entirely for request redaction but still requires ZK for
response verification.

### Redaction

Secret headers (`secretParams.headers`) are redacted via TLS 1.3 KeyUpdate — the client
rotates the symmetric key before and after the secret data, then only reveals keys for the
non-secret portions. The attestor never sees the plaintext API key. Public headers
(`params.headers`) appear in the proof's `parameters` field.

### Implications for Airnode integration

1. **3.85s per request is viable** for low-frequency authenticated data (not price feeds).
   With gnark, sub-1s total may be possible.
2. **The attestor is a separate service** — communicates over WebSocket, runs in Docker.
   AGPL license is avoided by process boundary. For production, Airnode would need a
   lightweight WebSocket client (not attestor-core).
3. **The proof format is compact** — identifier hash + signature + parameters JSON. Can be
   attached to Airnode responses alongside the existing EIP-191 signature.
4. **On-chain verification** is available via Reclaim's deployed Solidity contracts
   (`verifyProof()`) on Ethereum, Polygon, Arbitrum, BNB Chain, and others.
5. **The trust model shifts** from "trust the oracle's signing key" to "trust the attestor."
   When ChainAPI runs the attestor independently of the API provider, this provides
   meaningful third-party verification.
